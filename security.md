# SceneChat Security Architecture

**Team Resurgent / Darkone83**

This document details the full security model of SceneChat — from the initial DH handshake through per-packet encryption to credential storage. All cryptographic primitives are implemented from scratch in C89/MSVC2003 for the Xbox client and in Python for the server, with no reliance on TLS or external crypto libraries on the client side.

## Threat Model

SceneChat is designed for a **private trusted network** — a group of known users connecting to a self-hosted VPS. The security goals are:

- **Confidentiality** — message content, credentials, and room data must not be readable by passive observers on the network path between Xbox and server
- **Integrity** — packets must not be modifiable in transit without detection
- **Authentication** — users must prove knowledge of their password to log in; invalid credentials must be rejected
- **Replay resistance** — captured encrypted packets must not be replayable

SceneChat does **not** claim to defend against:
- A compromised VPS (the server holds the DB and sees decrypted traffic)
- Man-in-the-middle attacks on the DH handshake (no server certificate verification)
- Physical access to the Xbox hard drive (creds.dat is stored unencrypted)

## Protocol Overview

SceneChat uses its own binary protocol called **SCCP (SceneChat Chat Protocol)**. All communication after the DH handshake is encrypted with ChaCha20-Poly1305. There is no plaintext communication after the initial TCP connection is established — the DH messages themselves carry no sensitive data.

```
Xbox                                    Server
  |                                       |
  |--- TCP connect (port 8943) ---------->|
  |                                       |
  |<-- DH_INIT (0x01) -------------------|  DH params + server public key (plaintext)
  |--- DH_RESPONSE (0x02) -------------->|  Xbox public key (plaintext)
  |                                       |
  |    [both sides compute shared secret and derive session key via HKDF]
  |                                       |
  |--- REGISTER (0x03) [encrypted] ----->|
  |  or LOGIN (0x04) [encrypted] ------->|
  |<-- AUTH_OK (0x05) [encrypted] -------|
  |  or AUTH_FAIL (0x06) [encrypted] ----|
  |                                       |
  |<-- ROOM_LIST (0x07) [encrypted] -----|
  |--- JOIN_ROOM (0x08) [encrypted] ---->|
  |<-- ROOM_INFO (0x09) [encrypted] -----|
  |<-- HISTORY (0x0C) [encrypted] -------|
  |                                       |
  |--- MESSAGE (0x0A) [encrypted] ------>|  user sends message
  |<-- MSG_RECV (0x0B) [encrypted] ------|  server broadcasts to room
  |                                       |
  |--- PING (0x0E) [encrypted] --------->|  keepalive
  |<-- PONG (0x0F) [encrypted] ----------|
  |                                       |
  |--- DISCONNECT (0x10) [encrypted] --->|
```

## Packet Framing

### Plaintext packets (DH handshake only)

```
[2 bytes] total length (big-endian, includes type byte)
[1 byte]  packet type (SCCP_DH_INIT = 0x01, SCCP_DH_RESPONSE = 0x02)
[N bytes] payload
```

### Encrypted packets (all post-handshake traffic)

```
[2 bytes]  total length (big-endian)
[1 byte]   packet type
[12 bytes] nonce (random, generated fresh per packet via XNetRandom)
[N bytes]  ciphertext (ChaCha20 encrypted payload)
[16 bytes] Poly1305 authentication tag
```

The nonce is generated fresh for every single packet using `XNetRandom` on the Xbox side and `os.urandom` on the server. Nonce reuse would be catastrophic for ChaCha20-Poly1305 security — a unique random nonce per packet prevents this regardless of connection duration or message volume.

## DH Key Exchange

### Parameters

- **Key size:** 1024-bit (reduced from 2048-bit for Xbox 733MHz performance — 2048-bit took ~2 minutes, 1024-bit takes ~15 seconds)
- **Generator (g):** 2
- **Modulus (p):** 1024-bit safe prime generated by the server at startup using Python's `cryptography` library (`dh.generate_parameters(generator=2, key_size=1024)`)
- **Parameters:** regenerated each server start, transmitted to the Xbox in DER format inside `DH_INIT`

### Handshake

**Server side (`DH_INIT` — 0x01):**
1. Server generates DH parameters `(p, g)` and its own private/public key pair
2. Serialises `p` and `g` as a DER `DHParameter` sequence, server public key `sy` as a DER `SubjectPublicKeyInfo` bitstring
3. Sends both in the `DH_INIT` packet (plaintext — no sensitive data at this stage)

**Xbox side (`DH_RESPONSE` — 0x02):**
1. Parses the DER sequence to extract `p`, `g`, and `sy` (raw big-endian integers)
2. Generates its own private key `cx` — 1024-bit random value via `XNetRandom`
3. Computes client public key: `cpub = g^cx mod p`
4. Sends `cpub` as a raw 128-byte big-endian integer in `DH_RESPONSE`
5. Computes shared secret: `shared = sy^cx mod p`

**Server side (on receiving `DH_RESPONSE`):**
1. Extracts client public key `cpub` (128 bytes)
2. Computes shared secret: `shared = cpub^sx mod p` (where `sx` is server private key)

Both sides now hold the same `shared` — 128 bytes of big-endian integer.

### Bignum Implementation

The Xbox has no bignum library. A custom implementation in `sc_dh.cpp` provides:

- `bn_mul(r, a, b)` — schoolbook O(n²) byte-by-byte multiplication into a 256-byte result buffer
- `bn_reduce(r, a512, m)` — Barrett-style left-shift modular reduction over `BN_BYTES*8` iterations with a correcting final conditional subtract
- `bn_mulmod(r, a, b, m)` — multiply then reduce
- `bn_modexp(r, base, exp, m)` — square-and-multiply: iterates over every bit of `exp`, squaring and conditionally multiplying

All arithmetic operates on fixed-size `BN_BYTES` (128-byte) arrays in big-endian byte order. No heap allocation — all temporaries are stack-allocated fixed arrays.

## HKDF Key Derivation

The raw DH shared secret is not used directly as the session key. It passes through **HKDF-SHA256** (RFC 5869, extract-and-expand) to produce a well-distributed 32-byte session key.

```
PRK  = HMAC-SHA256(salt=zero32, IKM=shared_secret)
KEY  = HMAC-SHA256(PRK, info="scenechat-session" || 0x01)
```

`info` is the fixed byte string `b'scenechat-session'` — domain-separates this key from any other use of the same shared secret.

Both sides compute this independently after the DH exchange. The resulting 32-byte `session_key` is used for all subsequent ChaCha20-Poly1305 operations on that connection.

**Implementation:** `sc_net.cpp` contains a self-contained HMAC-SHA256 built on top of a hand-rolled SHA-256 block function, and HKDF built on top of HMAC-SHA256. No external crypto library is used on the Xbox client.

## ChaCha20-Poly1305 Encryption

### ChaCha20 (RFC 7539)

ChaCha20 is a stream cipher operating on a 512-bit (64-byte) state initialised as:

```
state[0..3]   = "expa", "nd 3", "2-by", "te k"   (constants)
state[4..11]  = session_key (32 bytes, 8 DWORD words, little-endian)
state[12]     = block counter (starts at 1 for message encryption, 0 for Poly key)
state[13..15] = nonce (12 bytes, 3 DWORD words, little-endian)
```

20 rounds of the quarter-round (QR) function:

```c
#define QR(a,b,c,d)       \
    a+=b; d^=a; d=ROL32(d,16); \
    c+=d; b^=c; b=ROL32(b,12); \
    a+=b; d^=a; d=ROL32(d, 8); \
    c+=d; b^=c; b=ROL32(b, 7)
```

Applied as 10 column rounds + 10 diagonal rounds per block. Output XORed with plaintext.

### Poly1305 MAC

Poly1305 operates over GF(2^130 - 5). The 32-byte Poly1305 key is derived by running ChaCha20 with counter=0 over 32 zero bytes:

```
poly_key = ChaCha20(session_key, nonce, counter=0, plaintext=zero32)[0:32]
```

The message is processed in 16-byte blocks. Each block is interpreted as a little-endian integer, a high bit of 1 appended (for full blocks), added to accumulator `h`, then multiplied by `r` modulo `2^130 - 5`. The final tag adds the `s` half of the Poly1305 key to the accumulator.

### AEAD Construction (RFC 7539)

```
poly_key       = ChaCha20(key, nonce, counter=0, zero32)
ciphertext     = ChaCha20(key, nonce, counter=1, plaintext)
tag            = Poly1305(poly_key, ciphertext || pad || len_aad(8) || len_ct(8))
```

AAD (Additional Authenticated Data) is empty in SCCP — only the ciphertext is authenticated. The length encoding matches RFC 7539: 8-byte little-endian AAD length (zero) followed by 8-byte little-endian ciphertext length.

Decryption verifies the Poly1305 tag before returning any plaintext. Tag mismatch causes immediate connection drop.

## Packet Types and Flow

All values are big-endian unless noted. Strings use a length-prefixed encoding:
- `string8` — 1-byte length prefix + UTF-8 bytes (max 255)
- `string16` — 2-byte length prefix + UTF-8 bytes (max 65535)

| Type | ID | Direction | Encrypted | Description |
|------|----|-----------|-----------|-------------|
| `DH_INIT`     | 0x01 | S→C | No  | DH params + server public key |
| `DH_RESPONSE` | 0x02 | C→S | No  | Xbox public key |
| `REGISTER`    | 0x03 | C→S | Yes | username + password |
| `LOGIN`       | 0x04 | C→S | Yes | username + password |
| `AUTH_OK`     | 0x05 | S→C | Yes | user_id + token + username |
| `AUTH_FAIL`   | 0x06 | S→C | Yes | error string |
| `ROOM_LIST`   | 0x07 | S→C | Yes | array of room_id + name + type |
| `JOIN_ROOM`   | 0x08 | C→S | Yes | room_id |
| `ROOM_INFO`   | 0x09 | S→C | Yes | room_id + name + member count |
| `MESSAGE`     | 0x0A | C→S | Yes | content string |
| `MSG_RECV`    | 0x0B | S→C | Yes | room_id + username + content + timestamp |
| `HISTORY`     | 0x0C | S→C | Yes | array of past messages |
| `ERROR`       | 0x0D | S→C | Yes | error string |
| `PING`        | 0x0E | C→S | Yes | keepalive |
| `PONG`        | 0x0F | S→C | Yes | keepalive response |
| `DISCONNECT`  | 0x10 | C→S | Yes | graceful disconnect |

## Credential Security

### Password Storage

Passwords are **never stored in plaintext**. On registration:

1. Server generates a random bcrypt salt
2. Computes `bcrypt(password, salt, rounds=12)`
3. Stores only the hash in the `users` table — the plaintext password is discarded immediately

bcrypt's work factor of 12 means ~250ms per verification attempt on modern hardware, making brute-force attacks computationally expensive.

### Password Transmission

Passwords travel inside the ChaCha20-Poly1305 encrypted tunnel exclusively. The cleartext password is never present on the wire. An observer capturing the TCP stream sees only the DH messages (no sensitive data) and encrypted ciphertext (computationally infeasible to decrypt without the session key).

### Credential Persistence on Xbox

After successful login the server returns a 128-character hex session token. The Xbox client stores this along with the username and password in `D:\creds.dat` for auto-login on next boot. This file is **not encrypted** — it is a flat binary file on the Xbox hard drive. Physical access to the Xbox drive gives access to stored credentials.

---

## Session Tokens

Post-login, the server issues a session token:

```python
token = secrets.token_hex(64)   # 64 random bytes = 128 hex characters
```

`secrets.token_hex` uses the OS CSPRNG (`/dev/urandom` on Linux). The token is stored in the `users` table alongside a `token_expiry` timestamp. Currently the token is used for auto-login — the Xbox sends username + password on reconnect rather than reusing the token for re-authentication, so the token serves primarily as a persistent identifier.

## Admin Security

### Panel Authentication

The Flask admin panel (`admin.py`, port 8950) uses:
- **Superadmin** — bcrypt hash hardcoded in source, never in the database
- **Admin/Moderator users** — bcrypt hashes in the `users` table, role column checked on login
- **Flask sessions** — server-side session with a 32-byte random `secret_key` generated at startup

Role hierarchy:
```
superadmin > admin > moderator > user
```

Moderators can: monitor chat, delete messages, ban/unban users.
Admins can: all of the above + role assignment.
Superadmins can: all of the above + delete users, create/delete rooms.

### Admin Message Bridge

Admin messages are sent via a minimal asyncio HTTP server on `localhost:8951` (not exposed externally). Flask POSTs to this endpoint; the asyncio handler injects the message directly into the server's `connected_clients` broadcast loop using the same `write_encrypted` path as normal messages. The admin message appears in chat as `[Admin]` highlighted in green.

Port 8951 binds to `127.0.0.1` only — it is not reachable from outside the server.

## Known Limitations

| Limitation | Impact | Notes |
|------------|--------|-------|
| 1024-bit DH | Below modern recommendations | Chosen for Xbox performance. NSA-class adversary could factor. Acceptable for private use. |
| No server verification | MITM possible on DH | No certificate pinning. Trusted network assumed. |
| Static DH parameters | Parameters reused across connections | `p` and `g` regenerated only on server restart, not per-connection |
| creds.dat unencrypted | Physical drive access exposes credentials | Acceptable for dedicated Xbox hardware |
| Empty AAD | Packet type not authenticated | Packet type byte is outside the AEAD envelope |
| No forward secrecy after key compromise | If session key is extracted from RAM, past traffic (if captured) is decryptable | Per-connection keys limit the blast radius |
| Admin panel HTTP | Flask runs plain HTTP on port 8950 | Acceptable behind a firewall/VPN; add nginx + TLS for public exposure |
